# 우리.zip 리빌드 계획

## 1. 서비스 리빌드 목적과 이유

### 서비스 개요

**우리.zip**은 가족 간의 소통을 영상으로 기록하고 공유하는 **Q&A 기반 가족 아카이빙 서비스**다.

바쁜 일상 속에서 가족 구성원들이 매주 주어진 질문에 영상으로 답변하고, 그 답변을 가족 단위로 모아 시간이 지나도 다시 꺼내볼 수 있는 디지털 아카이브를 만드는 것을 목표로 한다.

### 리빌드 목적

기존 팀 프로젝트는 서비스 아이디어와 핵심 기능의 가능성을 검증했다. 프론트엔드, 백엔드, AI 서버를 각각 로컬에서 연결해 시연 가능한 상태까지 구현했으나, 각 서버를 독립적으로 배포한 뒤에는 환경변수, CORS, 서버 간 URL, AI 서버 접근 경로 문제로 end-to-end 플로우가 안정적으로 동작하지 않았다.

이후 시연을 위해 일부 설정이 로컬 주소 기준으로 임시 고정되었고, 결과적으로 배포 링크 하나로 전체 서비스를 재현하기 어려운 상태가 되었다.

이번 리빌드는 기존 팀 프로젝트를 단순히 다시 만드는 것이 아니라, **로컬 중심 시연 구조를 제품형 모바일 서비스 아키텍처로 재설계하는 것**을 목표로 한다.

### 기존 프로젝트에서 이어받을 가치

- 가족 Q&A 기반 영상 아카이빙이라는 서비스 아이디어
- 가족 생성, 초대, 주차별 질문, 영상 답변, 아카이브라는 핵심 도메인
- 얼굴 정렬, 썸네일 생성, STT 요약이라는 AI 기능 방향성
- 팀 프로젝트를 통해 검증한 사용자 시나리오와 주요 화면 흐름

### 리빌드에서 해결할 문제

- 웹앱 중심 UX를 모바일 앱 중심 UX로 전환
- 배포 환경에서 프론트, 백엔드, AI 서버가 함께 동작하도록 구조화
- 영상 업로드와 AI 분석을 분리해 지연과 실패에 강한 흐름으로 변경
- 환경변수, 비밀값, 파일 저장, API 계약, 배포 구조를 명확히 문서화
- 기존 기능 중심 구현을 도메인 중심 구조로 재설계

### 포트폴리오에서 강조할 가치

- Expo React Native 기반 모바일 앱 전환
- 영상 업로드와 AI 분석을 분리한 비동기 처리 구조 설계
- AWS S3, CloudFront, SQS를 활용한 미디어 중심 서비스 아키텍처 경험
- 기존 팀 프로젝트의 운영/배포 한계를 분석하고 개선한 리빌드 사례
- 단순 CRUD가 아닌 모바일, AI, 클라우드 기반 서비스 설계 역량

## 2. 이전 서비스 구조 및 구현 범위 정리

### 이전 기술 스택

| 영역       | 기존 스택                                                                |
| ---------- | ------------------------------------------------------------------------ |
| Frontend   | React, Vite, JavaScript, React Router, Tailwind CSS                      |
| Backend    | Spring Boot, Java 17, MySQL, JWT, S3 연동                                |
| AI Server  | Flask, Google Vision, Gemini, MediaPipe, OpenCV                          |
| Deployment | Frontend: Vercel / Backend, AI: 배포 흔적은 있으나 현재 안정 재현 어려움 |
| Repository | FE, BE, AI 개별 레포 분리                                                |

### 이전 구현 목표

기존 서비스는 다음 기능을 목표로 했다.

- 가족 생성 및 초대 코드 기반 가입
- 주차별 질문 제공
- 사용자의 영상 답변 촬영 및 업로드
- AI 기반 얼굴 정렬 안내
- AI 기반 썸네일 생성
- STT 기반 영상 요약 및 제목 생성
- 가족 구성원별/주차별 영상 아카이브
- 일상 기록
- 쪽지함
- 마이페이지 및 접근성 기능

### 기존 구조의 문제점

#### 배포와 실행 재현성 문제

기존 프로젝트는 로컬 환경에서 프론트엔드, 백엔드, AI 서버를 연결해 시연 가능한 상태까지 구현했다. 그러나 세 서버를 각각 독립적으로 배포한 뒤에는 환경변수, CORS, 서버 간 URL, AI 서버 접근 경로 문제가 발생했고, 배포 환경에서 전체 플로우가 안정적으로 동작하지 않았다.

배포된 프론트 번들에는 `https://woorizip.site` 백엔드 URL 흔적이 있었으나, 현재 해당 서버는 응답하지 않는 상태다. 이후 시연을 위해 일부 설정이 로컬 주소 기준으로 임시 고정되었고, 결과적으로 배포 링크 하나로 전체 서비스를 재현하기 어려운 구조가 되었다.

리빌드에서는 앱, API 서버, AI worker, DB, S3 저장소의 책임을 명확히 나누고, 환경별 설정과 배포 구조를 처음부터 문서화한다.

#### 모바일 서비스와 웹앱의 불일치

우리.zip의 핵심 경험은 영상 촬영, 가족 알림, 모바일 접근성, 음성 안내에 가깝다. 하지만 기존 구현은 Vite 기반 웹앱이었기 때문에 네이티브 앱에서 기대되는 촬영 경험, 파일 처리, 푸시 알림, 모바일 내비게이션을 충분히 살리기 어려웠다.

리빌드에서는 웹앱을 단순히 옮기지 않고, Expo React Native 기반 모바일 앱으로 사용자 흐름과 화면 구조를 다시 설계한다.

#### 영상 업로드와 AI 분석 구조 문제

기존 구조에서는 영상 업로드 후 백엔드가 AI 서버에 썸네일/STT 분석을 요청하고, 분석이 끝날 때까지 응답을 기다리는 구조에 가까웠다.

이 방식은 다음 문제가 있다.

- AI 분석이 오래 걸리면 사용자 요청이 지연됨
- AI 서버 실패가 영상 업로드 실패로 번질 수 있음
- 여러 사용자가 동시에 업로드하면 백엔드 부하가 커짐
- 분석 상태를 명확히 보여주기 어려움

리빌드에서는 영상 업로드, 답변 등록, AI 분석, 결과 조회를 분리한다. 사용자는 업로드 완료 후 바로 다음 행동을 할 수 있고, AI 결과는 `ANALYZING`, `READY`, `FAILED` 상태로 추적한다.

#### 보안 및 운영 문제

AI 레포에는 `.env` 파일이 Git에 포함된 흔적이 있고, 서비스 계정 파일과 임시 파일 관리도 정리되어 있지 않았다. 또한 AI 서버는 CORS 전체 허용, API key 요청 전달 등 운영 환경에 적합하지 않은 구조가 있었다.

리빌드에서는 모든 비밀값을 환경변수 또는 secret manager로 분리하고, 레포에는 `.env.example`만 남긴다. API key는 클라이언트 요청으로 전달하지 않고 서버 환경에서만 관리한다.

### 포트폴리오에서 강조할 핵심 문장

> 기존 팀 캡스톤 프로젝트의 서비스 아이디어와 핵심 기능을 기반으로, 로컬 중심 시연 구조, 배포 환경 재현성, 웹앱 중심 UX, 동기식 AI 처리 구조의 한계를 분석했다. 이후 Expo 기반 모바일 앱, Express API, FastAPI AI worker, S3/CloudFront/SQS 기반 클라우드 아키텍처로 재설계하여 실제 배포 가능한 가족 영상 아카이빙 서비스로 리빌드했다.

## Assumptions

- 기존 팀 레포는 원본으로 보존하고, 개인 리빌드는 별도 레포에서 진행한다.
- 기존 코드는 그대로 복사하지 않고, 도메인/기능/AI 로직을 참고해 새 구조로 재구성한다.
- 초기 목표는 웹이 아니라 Expo 모바일 앱이다.
- ORM은 Prisma를 사용한다.
- 반려동물 숏츠와 쪽지함은 핵심 플로우 완성 이후 후순위로 진행한다.
- 기존 `woorizip.site` 도메인의 재사용 여부는 소유권과 팀 프로젝트와의 경계를 확인한 뒤 결정한다.
- AWS는 처음부터 모든 서비스를 쓰기보다 S3, CloudFront, SQS를 우선 도입하고, 배포 안정화 이후 App Runner/RDS/CloudWatch 범위를 확장한다.
